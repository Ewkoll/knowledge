GO中unsafe包

2020-05-13

最早很久之前的版本使用过unsafe包做string和[]byte的转换，当时感觉就是一些C里面的地址强制转换一样的东西。没怎么深入去看。
但是后续了解GO里面还有一些涉及到GC，还有一些内存扩张，转移一系列的东西，使用这个就不能像C++那样随便搞。

GO缓冲区溢出：

对于C的开发者来说，临时变量，如果出现缓冲区溢出问题，这个时候因为越界访问会导致函数的栈帧被覆盖，程序会崩溃，但是在GO中，目前还没有理解GO下层运行机制，以上的代码执行后，居然没有任何的异常。

这个结果是在第一次验证的时候是超越了个人理解。为啥GO这个程序即使这情况，都啥事没有的退出了，突然感觉对GO的知识还是太肤浅了。

本质还是在内存布局，推测是没有修改到运行函数栈帧对于的内存位置的执行代码，因此才没有出现问题。

字节序

字节序 就是多字节数据类型在内存中的存储顺序。可分为大端序，低地址端存放高位字节；小端序与之相反，低地址端存放低位字节。

数字305424265=0x12346789

小端模式
[0]: 137 : 0x89 ——– 低地址
[1]: 103 : 0x67
[2]: 52 : 0x34
[3]: 18 : 0x12
[5]: 0 : 0x00
[6]: 0 : 0x00
[7]: 0 : 0x00
[8]: 0 : 0x00 ——– 高地址

大端模式
[0]: 0 : 0x00 ——– 低地址
[1]: 0 : 0x00
[2]: 0 : 0x00
[3]: 0 : 0x00
[5]: 18 : 0x12
[6]: 52 : 0x34
[7]: 103 : 0x67
[8]: 137 : 0x89 ——– 高地址

小端模式，编译时候，如果需要从int32->int64只需要增加内存范围，不需要做移动操作。
大端模式，主要是用于网络数据包传输，可以直接按顺序读取数据。

encoding/binary可变长编码

binary包中PutVarint使用的是可变长编码的方式来处理数据。

通常一个字节8位存储数据最大就是8个1=0xFF=255，变长编码使用第一位标识是否需要读取下个字节的内容。

比如

88 = 0x58 小于 127 直接存储在一个字节即可，因为不需要读取后面的字节内容。存储个常规的数字一样的。

234 = 0xEA

1110 1010 大于了127就需要2个字节来存储对应的数据。

10000001 –> 表示需要读取下个字节。

01101010 –> 上个最后一个1和7位合起来。
