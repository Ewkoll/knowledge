C++

C++ 是一门多范式的通用编程语言。

多范式, C++支持面向过程的编程, 也支持面向对象的编程, 同时也支持范性编程. 据说最新版本支持函数式编程.

C++ 开发如果不是在大型公司有足够的应用领域真的会失业诶. 本身这门语言就很难, 而且, 涉及很多底层系统知识, 本身就具有复杂性问题, 而且在应用市场面临其它语言的竞争问题.

唯一还有的是高性能服务器领域和游戏领域了. 如果Rust语言成熟度上升到一定程度, 高性能服务器领域都没了.

OOP

C语言是面向过程的编程语言, C++是面向对象的编程语言. 可能会说C++更加复杂, 并且很难掌握, 这个主要是因为C++提供了更加复杂和多样的特性, 这些特性在使用和构建不当的时候, 很容易出现性能、编程复杂度问题, 因此C++的确需要很深的造诣去深入理解和使用才能使用的好.

封装

一个对象必定拥有属性和方法, 因此, 将一个事务对应的属性定义为对象的数据区, 将一个事务对应的能力定义为对象的方法的抽象过程就是封装.

封装的过程是通过程序的方式去描述世界物品特性的过程, 也是OOP的基础思想.

继承

通过继承父对象原来的属性和方法, 并扩展父对象的属性和方法的过程, 我们称之为继承, 这个过程我们可以新增父对象的属性和方法, 也可以重写父对象的方法.

继承的过程是通过程序的方式描述了世界物品属性扩展变更的过程, 这也是真实世界规则运行一部分. OOP也是来源于生活的.

多态

这里的多态专指运行时多态, 编译期间的多态个人觉得只是编译器绑定参数签名对应的函数地址功能.

在基类的函数前加上virtual关键字，在派生类中重写该函数，运行时将会根据对象的实际类型来调用相应的函数。如果对象类型是派生类，就调用派生类的函数；如果对象类型是基类，就调用基类的函数.

多态也代表了一个对象具体多种类型的能力, 对照世界, 水可以是气态、液态、固态、它在不同形态具有不同的属性和能力, 但是无论什么形态在分子层面他们构成都是H2O, 因此OOP的思想真的是对世界规则的深刻理解.

Class Or Struct

类

通过OOP思想对现实世界中具有共同特征事务的抽象, 简单的说就是一种抽象描述, 这个描述能够实例化出对象.

对象具体的一个类生成的实例, 在计算机世界中存在于应用的具体某个内存地址中.

结构体

结构体来源于C语言中一种数据结构的抽象.

本质上类和结构体在C++唯一具体的差异就是默认访问权限的不同, 在结构体中默认都是公有访问的, 在类中默认都是私有的, 在使用的时候, 如果是纯粹的数据对象使用结构体来描述, 如果是OOP思想的事物使用类来描述.

构造函数

一种特殊的方法, 因此本质上它是一个函数, 这个函数比较特殊的是因为当使用operator new操作符创建对象的时候会默认调用对应的构造函数.

一个类一定会有一个默认构造函数.

构造函数的主要用途用于初始化对象的属性信息, 执行初始化相关的逻辑代码.

种类:

  • 无参数默认构造函数
  • 一般参数构造函数
  • 拷贝构造函数
  • 等号运算符重载构造函数

特点:

  • 函数名与类名相同
  • 可以重载
  • 没有返回值
  • 编译器在没有定义时候自动生成此实现

补充:

构造函数不能添加virtual标志符.

1、空间角度, 虚函数对应一个虚函数表, 构造函数是虚函数, 如何通过vtable调用, 根本就不可能, 虚表都还没有实例化出来.
2、使用角度, 构造函数是创建对象时候自动调用的函数, 不可能通过父类指针去调用.

析构函数

析构函数(destructor) 与构造函数相反，当对象结束其生命周期，执行operator delete的时候, 默认会调用对象的析构函数.

特点:

  • 只能一个析构函数
  • 不能重载
  • 没有返回值
  • 编译器在没有定义时候自动生成此实现

虚析构函数

虚析构函数是为了解决基类的指针指向派生类对象，并用基类的指针删除派生类对象。

如果某个类不包含虚函数，那表示它将不作为一个基类来使用。当一个类不准备作为基类使用时，使析构函数为虚一般是个坏主意。因为它会为类增加一个虚函数表，使得对象的体积多一个指针大小，还有可能降低其可移植性。

这个移植性, 例如, 一个常规类型的对象, 当这个对象跨语言使用, 虚析构函数会新增一个虚表指针导致对象无法移植的问题.

或者当这个常规类对象, 在版本不同定义阶段一个版本定义了虚析构函数, 一个版本没有定义, 如果版本存在混合调用并使用对象传递参数的时候, 会导致异常.

所以基本的一条是：无故的声明虚析构函数和永远不去声明一样是错误的。实际上，很多人这样总结：当且仅当类里包含至少一个虚函数的时候才去声明虚析构函数。
抽象类是准备被用做基类的，基类必须要有一个虚析构函数，纯虚函数会产生抽象类，所以方法很简单：在想要成为抽象类的类里声明一个纯虚析构函数。

  • 务必为多态基类声明虚析构函数 (避免内存泄露);
  • 且只为多态基类声明虚析构函数 (减少内存占用，保护移植性);
  • 确保派生类的基类具有多态特性 (避免内存泄露).

虚函数

简单说, 当在类的函数加上virtual标志符, 这个函数就是虚函数, 存在虚函数, 那么在类的实例对象就会存在一个虚表

OOP多态也是通过虚函数来实现的.

多态是运行时执行不同的函数行为, 跟函数重载, 模版函数不同, 这些是编译期间决定调用函数地址.

纯虚函数

纯虚函数在虚函数的基础上在增加 “= 0” 表示此函数是纯虚函数, 表明:

1、这个类不能被实例化.
2、继承类必须实现此方法.

纯虚函数是为了描述, 当一个事务不适合直接实例化为一个真实对象的场景, 比如: 基类是动物, 继承类可以是猫, 狗等等, 但是真实世界中并不存在动物这么一个真实实体, 因此可以用纯虚函数来解决此类问题.

抽像类

抽像类只是一种特殊的类, 因定义了纯虚函数, 因此不能被直接实例化为一个对象.

因纯虚函数的特点, 只能定义接口, 不能包含实现, 因此抽像类要被使用, 必须被继承并实现对应的纯虚函数.

虚表

虚函数表(vtbl)
虚函数表指针(vptr)

当类存在虚函数的时候, 类对象都会有虚表, 用于存储每个对象多态行为, 保证当调用子对象的函数的时候, 如果是虚函数并且有实现, 将执行子对象的虚函数

特点:
1、父类有虚表, 所以继承的子类都会有虚表.
2、子类没有重写父类的虚函数的时候, 调用子类的虚函数, 等同于调用父类的虚函数.
3、虚表通常是函数指针数组.
4、虚标的顺序, 优先子类的虚函数, 然后是最顶层基类的虚函数.

对象切片(object slicing)

在值语义的情况下，派生类对象赋值给基类对象，派生类独有独有的数据会丢失。但是在指针、引用情况下，由于多态性，可以将派生类对象的指针赋值给基类指针，而不会导致对象切片。

常见的场景:

函数参数传递过程中, 如果一个参数是基类对象, 并且没有使用指针和引用传递, 直接使用值传递定义参数, 当我们把派生类对象传递到此函数参数的时候, 在函数内部会导致对象被切片成基类对象, 也就是说派生类独有的成员对象都被搞没了, 所以叫对象切片.

友元函数不是成员变量, 因此, 不能使用this指针.

RTTI

Run-Time Type Identification 运行时类类型识别

场景:

(父类) GameObject

(子类) HumanObject AnimalObject CarObject ChickenObject FirearmObject GunObject

参数传递父类对象, 通过RTTI识别并执行不同的逻辑代码(简单举例, 这个场景更多的会使用虚函数执行逻辑), 这种逻辑补丁代码是不良的代码设计缺陷.

使用基类指针或者引用检查实际派生类对象类型

RTTI的场景, 主要是在复杂多继承关系中切换对象指针或者引用的时候才有必须要使用RTTI, 大多时候简单的父子关系的指针偏移一般都是为0的, 在多继承关系通过RTTI来切换对象派生类, 用于访问派生类的成员.

C++存在虚函数, 才会存在多态性, 对于多态性对象, 编译期间无法确定对象的类型, 基类指针可以是指向任何派生类对象, 此时才需要RTTI去识别具体是那个对象.

操作符

  • typeid

typeid操作符, 通过typeid(对象).name()获取对象类型名称, typeid返回type_info的对象引用. type_info类没有实现默认构造, 复制构造, 赋值运算, 因此不允许直接创建此对象.

  • dynamic_cast

对于子类对象转换父类对象直接使用static_cast即可.

将基类指针或者对象安全的转换成派生类的指针或者引用, 执行此操作时候会访问运行时类型信息, 并检查转换是否有效. 并且当是指针的时候, 需要检查返回值判断是否为NULL, 这些代码都会消耗性能.

dynamic_cast必须是多态类对象指针或者引用才能使用此转换操作. 对于指针的转换, 如果转换失败返回的结果将会是NULL, 对于引用转换如果转换失败将会抛出异常.

性能测试

  • dynamic_cast is slow for anything but casting to the base type; that particular cast is optimized out
  • the inheritance level has a big impact on dynamic_cast
  • member variable + reinterpret_cast is the fastest reliable way to determine type; however, that has a lot higher maintenance overhead when coding

C++资源管理

堆

堆, 指的是动态内存分配区域

  • C++ new 和 delete 操作符操作的区域是free store[自由存储区本身是堆的子集]

  • malloc 和 free 操作的区域是heap[堆]

new 和 delete的底层实现是 malloc 和 free 来完成的. 因此动态内存分配区域就是说堆, 堆需要自己释放对应的内存.

飞行控制器和电信设备禁止使用动态内存分配. 就没有内存分配失败的问题… 我也是第一次知道有这么搞的.

  • 让内存管理器分配一个某个大小的内存块
  • 让内存管理器释放一个之前分配的内存块
  • 让内存管理器进行垃圾收集操作，寻找不再使用的内存块并予以释放

C++ 通常会做上面的操作 1 和 2。Java 会做上面的操作 1 和 3。而 Python 会做上面的操作 1、2、3。这是语言的特性和实现方式决定的。

栈

栈, 函数调用过程中本地变量和调用数据区域都属于栈. 后进先出, x86 在内的大部分计算机体系架构中，栈的增长方向是低地址.

栈, 太大的变量不能在栈空间中分配.

ulimit -s 返回8192 默认8M的栈空间.

栈一般向上表述, 来源于堆放盘子, 我们只能从上面拿和放.

因此在函数中声明的临时变量都是在栈中存储的.

每个调用函数都有自己的栈帧结构.

  • ebp 指向栈底 帧指针
  • esp 指向栈顶 栈指针
  • eip 寄存器指向下一条指向代码位置.

调用函数

  • 函数参数入栈
  • 返回地址入栈
  • 栈底指针ebp入栈
  • 修改ebp寄存器的值, esp的值修改为ebp的值

对于有构造函数和析构函数的非POD（Plain Old Data）对象, 编译器会在合适的位置插入对象构造和析构函数的调用.

编译器有一个重要特性, 就是自动调用析构函数, 即使在函数执行发生异常的情况, 这叫做栈展开 (stack unwinding).

静态存储区

程序编译、链接的时候确定的地址位置, 获取到程序基址、偏移量, 就可以直接计算出静态变量在内存中的位置.

全局变量、文件域静态变量、类成员静态变量在入口函数执行之前分配内存并初始化.
局部函数静态变量, 在此变量首次执行分配内存的时候, 执行初始化. 因此存在多线程并发问题.

非局部静态变量的初始化, 可以认为是线程安全的. 局部静态变量的初始化, 需要代码控制线程安全性.
需要避免使用局部静态变量锁, 在多线程环境下锁的构造存在问题.

RAII

Resource Acquisition Is Initialization 资源获取即初始化, 思想的来源主要是为了解决内存泄露, 资源泄露, 逻辑封装的问题.

RAII的思想本质是利用, C++语言特性局部对象自动销毁来控制资源的生命周期. (本质上就是堆空间的变量在离开执行区域会释放, 因此导致析构函数自动调用, 析构函数内部完成申请的内存资源或者句柄资源执行销毁来控制生命周期)

例如:

  • 临界区
  • 锁

当使用锁或者临界区的时候, 声明对象, 离开需要锁的范围的时候, 临时对象自动析构, 保证锁自动解开.

RAII对象也可以进行复制，但是拷贝的时候必须要一并复制它所管理的资源。因此在大部分场景我们为了避免忘记写资源复制的时候, 直接把RAII对象定义为noncopyable, 避免产生此错误.

一般不使用以下方式来解决资源复制问题, 最好直接定义资源是不允许拷贝

  • 通过boost::ptr_map, 非拷贝容器来保存资源.

内存泄露

Memory leak 动态分配的堆内存未释放或者无法释放, 内存泄漏缺陷具有隐蔽性、积累性的特征, 有些通常只会在特点逻辑场景才会触发, 并且积累隐患导致程序缓慢或者极端情况下出现崩溃问题.

分类:

  • 常发性内存泄露: 代码每次执行都会导致内存泄露.
  • 偶发性内存泄露: 只有特点的逻辑功能, 在特定逻辑条件下触发内存泄露.
  • 一次性内存泄露: 算法缺陷或逻辑缺陷导致程序总会导致一块内存泄露.
  • 隐式内存泄露: 运行期间一直使用内存, 不释放在程序关闭才释放, 导致内存无限扩张, 最终导致程序崩溃.

其它:

抽象类没有定义为虚析构函数, 在删除抽象类对象指针导致继承类对象没有执行析构函数, 导致逻辑和内存上的泄露是比较常见的一种编程缺陷错误.

检查方法:

  • 静态分析
  • 动态分析

通常简单程序可以通过Hook内存分配函数malloc和free函数, 通过跟踪块的申请释放情况来检查内存泄露的问题是否存在.

复杂的程序通常建议使用专业工具检查内存泄露问题.

跟内存泄露同样的是资源泄露, 在编程过程中一直申请系统非内存其它资源, 并不释放对应资源也是一直泄露, 最终会导致程序不稳定. 内存泄露问题更多的需要良好的设计结构和编码习惯来保证此问题尽可能少的发生.

对象大小

内存对齐，主要是为了性能和平台移植等因素，不同编译器有不同的默认对齐字节.

针对32位系统, 单个CPU周期读取32位的数据, 如果这个内存数据是非偶数范围, 这就导致需要2个CPU周期, 因此性能问题, 结构体需要进行内存对齐.

gcc中默认#pragma pack(4), 默认对齐字节是4个字节.

1、对齐原则, 第一个成员偏移量是0, 从0开始布局, 后续成员的当前偏移量必须是成员最大字节的整数倍. 比如某个成员是指针, 那么后续元素的对齐字节在64位系统下就是8个字节.
2、整个结构体的大小, 必须是对齐字节的整数倍.

柔性数组

C99定义了柔性布局, 如果结构体的最后一个成员是数组, 并且大小为0, 那么结构体的大小将不计入.

如果数组是有大小的, 将按照对齐规则执行对齐.

DCLP

个人见解, 为了保证多线程下单件初始化的安全性, 搞的这么复杂, 各种内存屏障, 指令重排问题, 为什么不在业务逻辑层面保证单件初始化, 在主线程中完成. 真TMD的操蛋.

volatile

volatile 关键字防止编译器对对应变量读写的优化[ 内存 -> 寄存器 ], 保证每次写入和读取都是通过原始对象内存地址中获取, 而不通过缓存.

由于 volatile 不能在多处理器的环境下确保多个线程能看到同样顺序的数据变化, java的此关键字保证顺序是因为java的编译器插入内存屏障指令代码.

内存排序

内存排序是指CPU访问主存时的顺序。可以是编译器在编译时产生，也可以是CPU在运行时产生。反映了内存操作重排序，乱序执行，从而充分利用不同内存的总线带宽。

内存屏障

同步屏障指令，它使得 CPU 或编译器在对内存进行操作的时候, 严格按照一定的顺序来执行, 也就是说在 memory barrier 之前的指令和 memory barrier 之后的指令不会由于系统优化等原因而导致乱序。

大多数现代计算机为了提高性能而采取乱序执行，这使得内存屏障成为必须。

语义上，内存屏障之前的所有写操作都要写入内存；内存屏障之后的读操作都可以获得同步屏障之前的写操作的结果。因此，对于敏感的程序块，写操作之后、读操作之前可以插入内存屏障。

内存排序

内存屏障, 分为编译期间的运行屏障, 通过在代码中嵌入汇编指令, 来防止编译器出现指令重排问题.
但是这个不能解决CPU运行的指令乱序问题, 因此需要原子操作来避免指令重排. 保证原子变量的读-修改-写指令中间不会插入其它指令.

atomic

原子对象 (atomic) 使用获得(acquire)、释放(release)语义

因为 atomic 变量的写操作缺省就是释放语义，读操作缺省就是获得语义, 本质上还是避免指令重拍, 避免指令重排, 也就是内存屏障.

  • 获得 (acquire) 是一个对内存的读操作，当前线程的任何后面的读写操作都不允许重排到这个操作的前面去。(y == 2)
  • 释放 (release) 是一个对内存的写操作，当前线程的任何前面的读写操作都不允许重排到这个操作的后面去。(y = 4)

原子量和互斥锁

  • 锁会导致阻塞, 性能比原子量低.
  • 原子量没有阻塞机制, 因此不能保护代码片段功能.
  • 锁的实现使用了原子量, 复合操作来完成锁的功能.
  • 内存屏障大约100-200CPU周期, 大约50纳秒, Linux上下文切换消耗1微妙的时间, 上下文切换是非常耗时的.

总线锁

每个处理器从自己缓存中读取数据, 执行对应操作在回写到主内存的顺序如果没有使用内存屏障, 执行顺序不可预测

总线锁, 就是CPU使用#LOCK信号, 其它CPU的请求会被阻塞, 该处理器能够独立使用共享内存数据.

CPU缓存行中内存区域数据在 #LOCK 操作期间被锁定, 缓存一致性机制会阻止被两个以上处理器同时修改缓存在内存区域的数据.

Light节点

2020-09-16
  1. 以太坊提供了 “fast”, “full”, “light” 三种数据同步模式
  2. 原来还有 –fast,–light 两个命令行选项，现在已经被 –syncmode 取代

当指定同步模式为 “fast” 时，以太坊节点会从网络同步所有的区块头，区块体以及状态数据，但不对区块中的交易进行重放，只会对区块中的数据进行校验。
当指定同步模式为”full”时，以太坊节点会从网络同步所有的区块头，区块体并重放区块中的交易以生成状态数据。
当指定同步模式为“light”时，以太坊节点仅从网络中同步所有区块头，不去同步区块体，也不去同步状态数据，仅在需要相应区块和状态数据时从网络上获取。

当不指定任何模式时，默认为 “fast” 模式。

为啥要提供三种模式

  1. 随着以太坊区块数据的越来越多，全节点同步，也就是“full”这种数据同步模式实在是太慢了，往往需要一周甚至更多时间。但以太坊网络又需要这些全节点去保存可追溯的历史数
  2. 运行轻节点，也就是“light”这种数据同步模式，因为只需要同步区块头数据，数据同步在几个小时就可以完成，但很多操作都需要这些轻节点去网络上其它全节点去请求数据，显然仅仅靠轻节点去组成区块链网络是不现实的。
  3. “fast” 这种同步模式就是“full”模式和“light”模式的一个折衷，通过直接从网络同步状态数据而不是通过重放交易生成状态数据的方式，节省了很多时间，一般来说机器配置合理的话 2～3 天就可以完成主网数据同步。

三种模式的安全性

毋庸置疑，全节点模式是最安全的，在这种模式下，所有区块和区块中的交易都会被逐一验证。
在 “fast” 这种数据同步模式下，比较靠近现在的区块和其中包含的交易也都会被逐一验证。但不会验证比较老的区块数据。
在 ”light“ 这种轻节点模式下，不会对区块和区块中的交易进行交易。但在需要处理特定区块相关交易或需要读取某个区块中相关的状态数据时，会通过区块头中的哈希值对区块和状态数据进行校验。

该用什么模式

自己搭节点，没有特别需求的话，使用默认的 ”fast“ 同步模式就好了。
如果要在树莓派等低配置设备中，可以使用 ”light“ 同步模式。使用”light“模式时，本地签名并发送交易一般没什么问题，但要获取区块数据或账户余额时，对网络情况就有更多的要求了。
如果特别注重安全性，并且想获取所有区块链历史数据，只能使用全节点模式了。

常见问题

问题： light 节点与 full 节点是如何交互的？
解答： 首先，light 节点会维护多个与 full 节点的 p2p 连接。然后，当 light 节点需要与 full 节点交互时，会将需要发送的请求放到一个请求队列中。light 节点会启动一个 goroutine 不断从请求队列里获取请求，然后从 p2p 节点列表里选一个当前最好用的节点，将请求发出去。请求得到的结果会发到本地数据库和缓存里。

结语

本文介绍了自己搭节点时需要注意的几种数据同步模式。其实很多 DApp，包括一些去中心化钱包，自己是不搭节点，而是依赖于第三方节点比如 Infura 的。
